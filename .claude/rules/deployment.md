# Deployment Strategy

This file describes how the project gets from the user's laptop to a
polished Streamlit app that recruiters can click from a CV link. Read
this before doing any deployment work.

## The target end state

The final deliverable for the interview is:

1. A **private GitHub repository** that holds all the code, data, and
   documentation during development.
2. A **Streamlit Community Cloud deployment** connected to that repo,
   providing a live URL like `https://ontario-fir-qa.streamlit.app` or
   similar.
3. A clean one-line addition to the user's CV: something like "Ontario
   FIR Data Quality Framework | github.com/USER/ontario-fir-qa |
   ontario-fir-qa.streamlit.app".
4. The repo flips from **private to public** only when the user
   explicitly confirms the code, commit history, and docs are clean.

## Why this stack

- **Local development** on the user's Mac gives Claude Code full access
  without the friction of a cloud IDE.
- **GitHub private** lets us iterate and make mistakes without a hiring
  manager seeing work-in-progress commits.
- **Streamlit Community Cloud** is free, deploys from GitHub
  automatically on every push, requires zero DevOps, and produces a
  clean sub-domain URL that looks professional on a CV.
- **Streamlit itself** is a Python-first framework that works well with
  pandas DataFrames, which is what the framework produces. The user is
  building Python muscle, so Streamlit reinforces that.

## Data in the repository

Data files live in `data/raw/{year}/schedule_{nn}.xlsx`. They ARE
committed to the repo (not gitignored) because:

1. Streamlit Community Cloud pulls from GitHub, so the app needs the
   data to be in the repo to run after deployment.
2. All FIR data is licensed under Open Government Licence Ontario,
   which explicitly permits publication.
3. Approximately 30MB of xlsx files is well within GitHub's comfort
   zone (the soft limit is 5GB per repo, the hard limit on individual
   files is 100MB).

If the repo ever grows past 500MB, the mitigation is to pre-process the
xlsx files into parquet files (10x smaller) at the root and gitignore
the raw xlsx. This is a Phase 5 optimization, not a current concern.

## Initial Git setup (one-time, before Phase 1 work)

These commands initialize the repo locally, create the first commit,
and push to a private GitHub repository. Run from the project root:

```bash
# Initialize git
git init
git branch -M main

# First commit: the Phase 0 state
git add .
git commit -m "Phase 0: single-year Schedule 22 QA framework"

# Create private repo on GitHub via the web UI, then:
git remote add origin git@github.com:USERNAME/ontario-fir-qa.git
git push -u origin main
```

The GitHub repo must be created **as private** initially. The user
should confirm this explicitly before the first push.

## Streamlit app structure

Streamlit Community Cloud expects a specific entry point. The
convention is a file named `streamlit_app.py` at the repo root. This
file should be a thin wrapper that imports the real app code from
`src/fir_qa/dashboard.py`.

```python
# streamlit_app.py
import sys
from pathlib import Path

# Make src/ importable when Streamlit Cloud runs this file
sys.path.insert(0, str(Path(__file__).parent / "src"))

from fir_qa.dashboard import main

if __name__ == "__main__":
    main()
```

The actual dashboard code lives in `src/fir_qa/dashboard.py`. This
separation keeps the entry point minimal and the real logic testable.

## Dashboard features by priority

Build the dashboard in this order. Each item should be a working
feature before moving to the next.

1. **Summary page**: loads all report CSVs, shows flag counts per rule,
   per severity, per year (when multi-year data is available). A
   top-level "there are N flags across M municipalities" banner.
2. **Municipality detail**: dropdown to select a municipality, then
   shows every flag for that municipality across all rules. This is
   the killer feature because it lets the interviewer say "let me see
   flags for Toronto" and get an immediate answer.
3. **Rule detail**: dropdown to select a rule, then shows every
   municipality flagged by that rule with the detail column expanded.
4. **Cross-year trends** (Phase 1+): for a selected municipality,
   shows levy and effective rate over time with the cross-year rule
   flags overlaid.
5. **Downloadable reports**: a button to download the full summary CSV
   and each flags CSV.
6. **About page**: static markdown page explaining the project, the
   domain context, the rule set, and the limitations. This is where
   the interviewer reads what you actually did.

The dashboard MUST NOT try to re-run rules on the fly at app startup.
It should load pre-computed report CSVs from `reports/`. The engine is
run as a preprocessing step before pushing to GitHub, and the CSVs are
committed alongside the code. Otherwise Streamlit Cloud cold-start
time becomes unacceptable.

## Deploying to Streamlit Community Cloud

1. Visit https://share.streamlit.io and sign in with GitHub.
2. Click "New app" and select the private `ontario-fir-qa` repo.
3. Set the entry point to `streamlit_app.py`.
4. Set the Python version to 3.11 or 3.12.
5. Deploy. First build takes 2 to 5 minutes.

After deployment the app will be available at a URL like
`https://ontario-fir-qa-USERNAME.streamlit.app`. The user can claim a
custom subdomain via the Streamlit Cloud settings.

## The flip from private to public

The repo stays private until all of these are true:

1. All rules are implemented and pass without errors.
2. The dashboard is working with all planned features.
3. The README is written and formatted (see README checklist below).
4. The commit history is either clean or has been squashed to a
   sensible sequence of meaningful commits.
5. No sensitive information is in the repo (no API keys, no personal
   data, no private notes).
6. The user has personally reviewed every file in the repo.
7. Optional: the methodology note is written.

Only then: set the repo to public via GitHub settings. Once public, the
Streamlit Cloud deployment automatically becomes available to anyone
with the URL.

## README checklist (before going public)

The README.md is the first thing a recruiter sees. It must include:

- A one-paragraph description of what the project does and why it
  exists (portfolio project for a specific government role, using
  real Ontario FIR data).
- A prominent link to the live Streamlit app.
- A "Key findings" section with 2 or 3 bullet points of interesting
  anomalies found (e.g., Brant County $12.9M excess).
- A short "How it works" section with a code block showing how to run
  the engine locally.
- A "Rules implemented" section listing all rules with a one-line
  description each.
- A "Limitations and future work" section (honesty impresses more
  than hype).
- Attribution: "Uses data from the Ontario Financial Information Return
  published by the Ministry of Municipal Affairs and Housing under the
  Open Government Licence Ontario."
- A screenshot or two of the dashboard.

## Deployment anti-patterns to avoid

1. **Committing secrets**: never commit API keys, .env files, or
   personal notes. The .gitignore should cover this but do a manual
   audit before the first push.
2. **Large binary files**: if you add images or screenshots to the
   repo, keep them under 1MB each and use a single `docs/images/`
   directory.
3. **Broken deployments**: do not push changes that break the live
   Streamlit app once it is linked from the CV. Test locally first
   with `streamlit run streamlit_app.py`.
4. **Computing on the fly**: the deployed app should not re-run the
   rule engine at startup. Pre-compute reports, commit the CSVs, and
   have the app load the CSVs.
5. **Forgetting to squash work-in-progress commits**: before going
   public, review the commit history. If there are 40 commits titled
   "wip" or "fix", consider squashing them into a smaller number of
   meaningful commits via interactive rebase. This is not strictly
   necessary but looks more professional.
