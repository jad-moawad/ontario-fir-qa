# Ontario FIR Data Quality Framework

Portfolio project for a job application to the Ontario Ministry of Finance,
Property Tax Services Partnerships Branch, Data and Quality Assurance Unit
(Senior Data and Quality Analyst, Job ID 242142, closes April 16 2026). The
framework loads Ontario Financial Information Return (FIR) Schedule 22 files,
runs domain-aware quality assurance rules, and reports flagged anomalies.

The pitch at the interview is "I built a small version of what I imagine your
team does, using the data your ministry actually collects."

4. For LaTeX output, always provide the .tex source alongside the PDF.
5. Industry CVs and resumes are one page maximum unless explicitly requested
   otherwise. Academic CVs can go longer.

## Quick start

```bash
pip install -r requirements.txt
PYTHONPATH=src python3 -m fir_qa.engine data/raw/2023/schedule_22.xlsx reports/2023
```

Expected output on the 2023 file: 0 flags for R01 through R04, 9 for R05, 95
for R06, 1 for R07. If these numbers match, the project is healthy.

## Code conventions

- Python 3.10+. Pandas for data, standard library for everything else.
- Each rule function is decorated with `@_rule_meta(rule_id, rule_name,
  severity)` which attaches metadata as function attributes. This lets the
  summary builder know the rule's name and severity even when the rule
  returns zero flags.
- Rules return a DataFrame of flagged rows with three metadata columns
  prepended (rule_id, rule_name, severity), plus rule-specific columns and a
  `detail` column explaining the flag in plain English.
- Tolerances are explicit. $1 for absolute dollar comparisons, 0.5% for
  relative comparisons, 1e-7 for tax rate comparisons.
- The loader handles the FIR xlsx quirks (header on row 5, unnamed ID
  column, embedded newlines in column names, mixed numeric types). Do not
  reinvent this. If adding support for another schedule, extend the loader
  module, do not duplicate the logic.

## What NOT to do

- Do not add rules that flag false positives. A noisy rule is worse than no
  rule because it trains the user to ignore the report.
- Do not remove Rule 06 even though it has a high flag count. It is
  downgraded to warning severity with a documented limitation, and keeping
  it demonstrates awareness of Ontario's graduated tax rate provisions.
- Do not reconcile against Total sheet line 9201. Use line 9299. Line 9201
  can have multiple rows per municipality when the municipality is split
  into levy areas (e.g. Frontenac Islands has Howe Island and Wolfe Island).
- Do not assume residential education tax rate is always 0.00153. It is
  only 0.00153 for RTC/RTQ code RT in lower-tier and single-tier
  municipalities. Upper-tier rows are NaN (UT municipalities don't levy
  education taxes directly). Sub-classes like R1 have their own rates.
- Do not hardcode file paths. The engine takes source and output as CLI
  arguments.
- Do not skip verifying your changes by re-running the engine. Every change
  to rules.py or loader.py must be followed by a clean engine run.
